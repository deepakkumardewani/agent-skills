# Impeccable — Project Design Context

> Persistent design context for this project. Loaded by `/impeccable craft` and related sub-commands so generated code stays on-brand without re-stating the direction each time.

Authoritative sources (read these for details, not this file):

- `SPEC.md` — what we're building, acceptance criteria, boundaries
- `DESIGN.md` — design tokens (colors, type, spacing, motion, components) + design principles + 5-minute review checklist
- `CLAUDE.md` — repo conventions, stack, Astro patterns, banned phrases

## Register

**product** — this is a docs/utility site. Earned familiarity is the bar: fluent users of Linear / Vercel / Stripe docs should trust the affordances. We diverge from those defaults in **color and typography** (warm amber + Manrope, not cool indigo + Inter) and **tone** (personal tribute, not corporate marketing) — but the interaction patterns and IA are conventional dev-docs.

## Design Context

### Users

Two overlapping audiences:

1. **AI-coding-tool users** (Claude Code, Codex, Cursor, etc.) doing task-driven skill discovery. Context: opens the site mid-work, looking for "is there a skill for X?" — wants to find it in ≤3 clicks and copy the trigger command.
2. **Developers new to the ADLC framework** who learn the lifecycle (Define → Plan → Build → Test → Review → Simplify → Ship) just by browsing the sidebar IA.

Both groups are senior-ish developers. Both expect dev-docs conventions (sidebar + content, `Cmd/Ctrl+K` search, code-block copy, dark mode). Neither needs hand-holding.

Job to be done: *"Help me find the right skill for what I'm doing right now, show me what it does, and let me copy the trigger command."*

### Brand Personality

Three words: **considered, generous, modest.**

Voice: warm, plainspoken, developer-honest. Like a senior engineer explaining a tool they've used for years — no marketing varnish, no hype, no condescension.

Emotional goals: trust (this is somebody's careful work), curiosity (the framework is worth understanding), respect (the author Addy is being honored without being deified).

Explicit non-goals: futurism, luxury, playfulness, corporate scale, urgency, "next-generation" hype.

### Aesthetic Direction

**Modern dev-docs with warmth** — diverges from the 2025 default (cool grays + Inter + indigo) by choosing warm stone neutrals + Manrope + a single amber accent.

**Visual references** (for *principles*, never for direct copy):

- *Stripe docs* — typography discipline, content hierarchy.
- *Notion docs* — restraint, generous whitespace, plainspoken voice.
- *Anthropic docs* — calm, considered, premium without being luxurious.

**Explicit anti-references** (do NOT visually emulate):

- *Linear / Vercel / shadcn docs* — sharing their aesthetic makes us look like an AI template.
- *Impeccable's site* — cream parchment, hairline serif, marbled gold; their editorial register is locked and copying it dilutes both projects. We are *modern dev-docs*, not editorial.
- *Generic SaaS landing pages* — gradient hero meshes, three-CTA rows, stock photos, "trusted by" logo marquees.

**Theme:** dark mode + light mode, both first-class. Defaults to system preference. Light is *warm off-white* (Stone 50, `#FAFAF9`), dark is *warm dark* (Stone 950, `#0C0A09`) — never pure white or pure black, never blue-cold dark.

**Single accent:** Amber 600 (`#D97706`). Reserved for focal elements: primary CTA, active sidebar item, focus ring, links. Never decorative.

**Phase chip palette:** 8 hues (stone, amber, orange, rose, emerald, violet, sky, lime) mapped to the lifecycle groups. Subtle (100-level) backgrounds with 700-level text.

**Typography:** Manrope for all UI + reading text (warmer humanist sans, chosen over Inter for distinctiveness). JetBrains Mono for code.

**Corners:** soft — 8px DEFAULT, 12px for cards, 16px for elevated cards, `full` for chips. Not sharp (Linear), not pillowy (consumer).

**Depth:** structural, not theatrical. Borders + tonal lift carry hierarchy. Shadows are barely there (`xs`/`sm`) except on the search dialog (`lg`) when modal-open.

### Design Principles

The five rules that override every other instinct. When in doubt, run a design decision against these in order:

1. **The one feeling first.** The hero of every page exists to produce *one* feeling: *"Somebody built this carefully. Takes its work seriously. Doesn't take itself too seriously."* If a design choice doesn't actively support that, cut it.

2. **One focal element per section.** In any viewport, the eye lands on exactly one thing first. Build hierarchy with size → weight → contrast → space, in that order. `display-lg` appears at most once per page. The 25%-zoom test: at 25% browser zoom, the focal element should still be obviously focal.

3. **Every element defends its existence.** Default to fewer sections, fewer words, fewer affordances. The landing has four sections (hero + ADLC + browse-by-phase + footer) — never five. If a section can be deleted without loss, delete it.

4. **Whitespace as confidence.** Start spacious (96px section margins, 68ch prose measure). Never tighten in a later polish pass. Whitespace says *"we know what matters."* Edge-to-edge density says *"we don't trust the user to find the thing."*

5. **Micro-interactions are the memory.** Every interactive element has hover, focus-visible, active, disabled, loading, empty, and error states designed up front. Transitions 140–200ms ease-out for hover; instant for active/pressed. `prefers-reduced-motion` always honored.

### Accessibility

- **WCAG 2.1 AA** baseline, enforced via `axe-core` in Playwright e2e (per SPEC §5).
- **Lighthouse Accessibility ≥95** on landing + a representative skill page (SPEC acceptance criterion).
- **Color contrast:** all body text ≥4.5:1, all large text + UI ≥3:1. Amber 600 on white = 4.6:1 (passes); amber 700 used for body links.
- **Keyboard-first:** every interactive element reachable + operable by keyboard, with visible focus rings (3px amber, never `outline: none` without replacement).
- **Reduced motion:** `prefers-reduced-motion: reduce` swaps animations for instant token swaps. No parallax, no scroll-jacking, no auto-playing motion.
- **No icon-only buttons without `aria-label`.** No color-only signals — phase chips include text labels.
- **Screen-reader-friendly markdown rendering:** prose uses semantic h1/h2/h3, lists, blockquotes — never `<div>`-soup styled to look like headings.

## What this context tells future runs

When `/impeccable craft` or any sub-command runs in this project, it should:

- Default to **`.astro` components** for static surfaces; reach for client islands only when interactivity demands it.
- Use **CSS custom properties from `tokens.css`** for every styled value. No raw hex/rgb/px in components.
- Treat the **8-color phase palette** as the only multi-hue family. Don't introduce decorative greens, blues, or pinks outside it.
- Apply the **banned-phrase list** from CLAUDE.md to user-facing copy. Synced SKILL.md bodies are exempt.
- Run the **5-minute review checklist** in DESIGN.md §9 before declaring a UI task complete.
- Never propose Tailwind, shadcn, MUI, Chakra, or any UI framework. Hand-rolled CSS only.
- Never frame the site as official, Anthropic-affiliated, or as anything other than a personal tribute.

When in conflict: SPEC > DESIGN > CLAUDE.md > this file > model defaults.
